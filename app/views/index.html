R10 iterations

1.1 Making it clear that standard recall includes indeterminate sentences

Some use comment to aks wether indeterminate recalls would have their own category, as in current service they have their own recally type optin. In new service, expectation is for user to chose standard, so we have modified the content to make that clearer.

  • Recall type

1.2 Sending users to review remand and tagged bail after recall

We previously had links to the adjustments service from the recall confirmation screen, we wanted to make this more specific to encourage users to go to adjustments to manage remand and tagged bail until we accomodate that within most-MVP work.

  • Confirmation screen

1.3 Making the actions on the manual journey clearer

Despite this page testing well, we wanted to remove any potential doubt or confusion about what we need the user to do on this page.

  • Select all cases that had an active sentence

2. Delete a recall

We are planning to allow user to delete recalls in the event where something has been recorded incorrectly and would not suffice to just edit the recall. For now, we would only allow for the most recent recall to be deleted, to avoid any unintended deletion of data which underpins active recalls.

  • Are you sure you want to delete this recall?

3. Displaying non-recallable sentences

Designs for how we plan to display sentence types which cannot be recalled (like fines) when recording a recall. Where court cases only have sentenced which cannot be recalled, we may hide the entire court case, but we need to consider how these are displayed when there is a mix of sentence types.

  • Select all cases that had an active sentence

4. Displaying sentence date

Users have previously requested that we display sentence date in the journey to allow them to better work out when the sentetence ended to know if it was active whent he person was recalled.

  • Select all cases that had an active sentence (manual journey)
  • Check sentences and offences (manual journey)
  • Check sentences and offences (happy path journey)

Where a recall would result in UAL, this must not overlap with any other adjustment. For existing recall UAL, we simply merge but for any other type we need the user to either modify the recall dates or review the existing adjustments.

3. Making it clearer which cases to select on the unsupported journey

R7 showed mixed results when testing this page. We agreed to tweak the content of the page to try and make it clearer that the user should select cases with active sentences.

  • Select cases (unsupported journey)

4. Allowing user to select any recall type

Previously we filtered the list of recall types and the user would not be able to choose an incorrect recall type. Previous rounds of research showed that sometimes users will recieve a revocation order from PPCS with a recall type which doesn't match the sentence type and length rules. The user may question this PPCS but it is not always corrected by PPCS. We need to allow the user a way of selecting any recall type whilst still highlighting when a recall type might be incorrect.

1.1 Improvements to the 'select the relevant offence information' page

Adding overall effective SLED to manual sentence selection page

Bringing the unhappy path design into line with happy path, we will surface the overall effectvie SLED on the 'select sentence' page to help people with choosing the appropriate sentences.

Requiring the user to select cases instead of sentences

R6 showed that users were confused by having to select relevant sentences, some selected all sentences just to be sure that they wouldn't be lost. After consideration around accuracy conversations with policy (still undecided). We are planning to test a simplified version of the manual selection where the user selects the entire case, this is intended to make the choice much simpler for the user.

  • Select sentence and offence information page

1.2 Improvements to the 'check sentence and offence information' page

Displaying both sentence and conviction dates to sentence cards

Users have expressed a need to see both sentence and conviction dates for the purposes of reviewing sentence information in recalls.

Modifying court case headings to add last warrant date

We've created a new pattern library for surfacing court case information, with that, a new naming convention for helping people identify a court case by adding in the date of last appearance.

  • Check sentence and offence information page

2.2 New way of displaying latest calculation dates on CRDS

A new design for showing the latest calculation on the release dates and calculation landing page. This design adds a summary list to allow more space to surface information on the source of calculation (particularly improtant where dates are entered on CRDS manually from a paper calculation)

  • Release dates and calculations landing page - source manual calc
  • Release dates and calculations landing page - source CRDS
  • Release dates and calculations landing page - source nomis
  • Calculation details - source manual calc
  • Calculation details - source CRDS
  • Calculation details - source NOMIS

2.4 Unhappy path - editing UAL overlapping UAL periods

For when there is UAL of the recall dates already existing on the service, we will automatically associate any overlap and then inform the user of what is happening.

1. New entry point to record a recall

From R2 we understood that user expected release dates to be calculated at the same time as recording a recall. Because of this, we want to test brinign recalls and CRDS closer together. In this version recall still has its own entry point, however it now lives on the release dates and calculations landing page. This also enable us to automaticall choose the calculation reason and take the user straight into the recalls journey.

  • Record a recall

2. Update 'choose a recall type'

We are able to use sentence information and the dates provided by the user to narrow down or infer the recall type for the user. We want to understand how helpful this could be and wether there are scenarios where the recall type on the revokation order is different to what the system suggests. Our designs use a single question "is this fixed term?" which should allow us to either infer the recall type or provide a small list of options for the user to then choose from. If we are able to identify the recall type, we insted will show a page which aims to communicate this and allows the user to confirm.

  • 'Is this a fixed term recall?' question page
  • 'Choose a recall type' options page
  • 'Confirm recall type' page
